## Strategic Decisions

### 1. NATS Adoption for Distributed Orchestration
- **Decision**: Adopt NATS as the messaging backbone for multi-node agent coordination.
- **Rationale**: Context explosion in single-agent dispatch is the primary scaling bottleneck. NATS enables pub/sub event routing across distributed Shikki nodes without centralized broker overhead.
- **Status**: Foundation complete (5 branches). Integration with ShikkiKernel is next.

### 2. Kernel Service Decomposition
- **Decision**: Decompose monolithic ShikkiKernel into discrete services (wake signals, escalation, scheduling).
- **Rationale**: Tighter service boundaries enable independent testing, NATS-based communication, and hot-swap upgrades.
- **Status**: Decomposition landed on `feature/kernel-decomposition-v2`.

### 3. Zero-Downtime Upgrade Architecture
- **Decision**: Spec written for rolling upgrades without service interruption.
- **Rationale**: Production Shikki instances (native binaries + systemd) must upgrade without dropping in-flight agent tasks.
- **Status**: Spec complete. Implementation depends on NATS node discovery (now delivered).

### 4. Wave-Based Dispatch Throttling
- **Decision**: Cap parallel agent dispatch at 6 per wave with 5-minute stagger.
- **Rationale**: API rate limits and context window pressure require controlled parallelism.
- **Status**: Operational. dispatch-resilience spec formalizes the pattern.
