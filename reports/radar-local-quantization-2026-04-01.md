## A) llama.cpp IQ Quants — Importance Matrix Quantization

### What Are IQ Quants?

IQ (Importance-matrix Quantized) types use an importance matrix (`imatrix`) to selectively assign precision to weights based on their impact on model quality. Weights that matter more get more bits. This produces dramatically better quality at extreme compression levels compared to uniform quantization.

The importance matrix is generated by running representative text through the model with `llama-imatrix`. It measures which weights contribute most to output quality, then the quantizer uses this data to allocate bits non-uniformly.

### Can You Run 70B at IQ2 on 64GB?

**Yes, but barely.** A 70B model at IQ2_M (~2.93 bpw) would be approximately 26-28 GiB. That fits in 64GB with ~36GB headroom for KV cache and system. However:

- Inference speed will be slow for 70B on CPU/Metal (~3-8 t/s for generation)
- Quality at IQ2 on 70B is mediocre — you lose too much for reliable code generation
- A 32B model at Q5_K_M (23.3 GiB) will produce better code than a 70B at IQ2_M
- **Recommendation**: 32B at Q5_K_M or Q6_K is the sweet spot for 64GB

### IQ vs K-Quant: Which Is Better?

At the same bits-per-weight, IQ quants produce better quality than K-quants because of the importance matrix. The tradeoff: IQ quants are slightly slower due to more complex dequantization. On Apple Silicon Metal, IQ quants are fully supported but marked as slower than K-quants of comparable size in the llama.cpp feature matrix.

**Practical rule**: Use IQ quants when you need to squeeze into a tight memory budget. Use K-quants when you have room and want maximum inference speed.

## B) MLX Native Quantization

### What Does `mlx_lm.convert` Support?

MLX LM (now at [ml-explore/mlx-lm](https://github.com/ml-explore/mlx-lm)) supports:

| Bit Level | Supported | Notes |
|-----------|-----------|-------|
| 2-bit | Yes | Via `mixed_2_6` recipe (mixed 2-bit + 6-bit) |
| 3-bit | Yes | Via `mixed_3_4` and `mixed_3_6` recipes |
| 4-bit | Yes | Default quantization, most widely used |
| 6-bit | Yes | Higher quality, used in mixed recipes |
| 8-bit | Yes | Near-lossless |

**Quantization modes**: `affine` (default), `mxfp4`, `nvfp4`, `mxfp8`

**Mixed-bit recipes** (the interesting part):
- `mixed_2_6` — 2-bit for unimportant layers, 6-bit for critical ones
- `mixed_3_4` — 3-bit base, 4-bit for important layers
- `mixed_3_6` — 3-bit base, 6-bit for important layers
- `mixed_4_6` — 4-bit base, 6-bit for important layers

**Command**:
```bash
# Standard 4-bit quantization
mlx_lm.convert --model Qwen/Qwen2.5-Coder-32B-Instruct -q

# Custom bits + group size
mlx_lm.convert --model Qwen/Qwen2.5-Coder-32B-Instruct -q --q-bits 3 --q-group-size 64

# Mixed-bit recipe
mlx_lm.convert --model Qwen/Qwen2.5-Coder-32B-Instruct -q --q-predefined mixed_3_6
```

### MLX vs GGUF Comparison

| Aspect | MLX | GGUF (llama.cpp) |
|--------|-----|------------------|
| **Target** | Apple Silicon only (Metal) | Cross-platform (CPU, Metal, CUDA, Vulkan) |
| **Format** | SafeTensors (`.safetensors`) | `.gguf` single file |
| **Quantization types** | 4 recipes + custom bits | 20+ types (IQ1-IQ4, Q2-Q8) |
| **imatrix support** | No (uses mixed-bit recipes instead) | Yes — per-weight importance |
| **Metal optimization** | Native, first-class | Supported, well-optimized |
| **Inference speed** | Generally faster on Apple Silicon | Slightly slower on Metal, faster on CPU |
| **Ecosystem** | Growing (mlx-community on HF) | Mature (thousands of GGUF models) |
| **Speculative decoding** | Supported | Supported |

**Bottom line**: MLX is faster for Apple Silicon inference but has fewer quantization options. GGUF has the richest quantization ecosystem. For LM Studio (your setup), GGUF is the native format.

## C) BitNet 1-bit — Microsoft's Ternary LLMs

### What Is BitNet b1.58?

BitNet b1.58 uses ternary weights: each weight is -1, 0, or +1 (1.58 bits per weight). This is NOT post-training quantization — models must be **trained from scratch** in this format. The inference engine (`bitnet.cpp`) uses specialized kernels that exploit the ternary structure.

### Current Status (March 2026)

| Aspect | Status |
|--------|--------|
| **Maturity** | Production-ready inference (bitnet.cpp v1.0, Oct 2024) |
| **Stars** | 36.9k on GitHub |
| **Apple Silicon** | Fully supported, demo on M2 shows 1.37-5.07x speedup |
| **CPU performance** | ARM: 1.37-5.07x speedup, 55-70% energy reduction |
| **GPU support** | Available (May 2025) |
| **Latest update** | Jan 2026 — CPU inference optimization with parallel kernels |

### Available Models

| Model | Parameters | Notes |
|-------|-----------|-------|
| BitNet-b1.58-2B-4T | 2.4B | Official Microsoft model |
| bitnet_b1_58-large | 0.7B | Community |
| bitnet_b1_58-3B | 3.3B | Community |
| Llama3-8B-1.58 | 8.0B | Community |
| Falcon3 Family | 1B-10B | Community |

### Can Shikki Run a 1-bit 70B on Apple Silicon?

**No 70B model exists.** The largest available is 10B. Microsoft claims a theoretical 100B model could run on a single CPU at 5-7 tokens/sec (human reading speed), but no one has trained one yet.

**The fundamental limitation**: BitNet requires training from scratch. You cannot take an existing 70B model and convert it to BitNet. Someone (Microsoft, Meta, or a large lab) needs to invest the compute to train a 70B BitNet model. As of March 2026, no one has.

### Verdict for Shikki

**PARK for now.** BitNet is the most exciting efficiency research but lacks the model ecosystem for practical code generation. The 2.4B official model is too small for serious coding tasks. Watch for:
- A BitNet model > 30B parameters
- BitNet fine-tuning support (train a coding model from a base BitNet)
- Apple MLX integration with BitNet kernels

### Winner: Qwen2.5-Coder-32B-Instruct

**Why it wins:**
1. **SOTA open-source coding** — matches GPT-4o on code generation benchmarks (Aider: 73.7, McEval: 65.9, MdEval: 75.2). SOTA on 5 FIM benchmarks.
2. **32B dense fits perfectly on 64GB** — at Q5_K_M (23.3 GiB) or Q6_K (26.9 GiB) with plenty of headroom
3. **128K context** — long enough for large file analysis with YaRN scaling
4. **116 quantized variants on HuggingFace** — mature GGUF ecosystem
5. **LM Studio native support** — works with your existing `http://127.0.0.1:1234` setup

### Recommended Configuration for 64GB M2 Max

| Config | Quant | Size | Quality | Speed (est.) | Use Case |
|--------|-------|------|---------|-------------|----------|
| **Best quality** | Q6_K | 26.9 GB | Very high, near-perfect | ~15-20 t/s gen | Primary coding model |
| **Best balance** | Q5_K_M | 23.3 GB | High | ~18-25 t/s gen | Primary if you need headroom |

**Recommended**: `Q5_K_M` or `Q6_K`. On a 64GB M2 Max, Q6_K (26.9 GB) leaves 37GB for KV cache, system, and Xcode/simulator. This is the sweet spot.
