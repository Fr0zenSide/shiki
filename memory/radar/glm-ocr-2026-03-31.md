## What Is It

GLM-OCR is a multimodal OCR model + SDK for complex document understanding, built by ZhipuAI (the GLM/ChatGLM team). It uses a vision-language architecture:

- **CogViT** visual encoder (pre-trained on large-scale image-text data)
- Lightweight cross-modal connector with token downsampling
- **GLM-0.5B** language decoder (only 0.9B parameters total)
- **Multi-Token Prediction (MTP)** loss + stable full-task reinforcement learning
- Two-stage pipeline: **PP-DocLayout-V3** layout analysis + parallel region OCR

**Key claim**: #1 on OmniDocBench V1.5 with score 94.62. State-of-the-art on formula recognition, table recognition, and information extraction benchmarks.

## Architecture

```
Input (image/PDF)
    |
    v
[PP-DocLayout-V3] -- layout detection (25 region types)
    |
    v
[Parallel Region Cropping] -- bbox + optional polygon masks
    |
    v
[GLM-OCR VLM] -- per-region OCR via vLLM/SGLang/Ollama/MLX/MaaS
    |
    v
[ResultFormatter] -- JSON + Markdown output
```

The SDK is a **pipeline orchestrator** (not just a model wrapper). It handles:
- PDF-to-image conversion (PyMuPDF)
- Layout detection with PP-DocLayout-V3 (25 label types: text, table, formula, seal, chart, etc.)
- Parallel region recognition via thread pool (up to 32 workers)
- Post-processing: formula merging, text block merging, bullet formatting
- Output in JSON (structured with bbox) and Markdown

---

## Languages & Content Types

- **Text**: General text, vertical text, seals, handwriting
- **Tables**: Complex multi-row/column tables -> Markdown
- **Formulas**: Display + inline formulas -> LaTeX
- **Code**: Code blocks
- **Documents**: Headers, footers, page numbers, references, footnotes
- **Languages**: Primarily CJK + English. Model trained on large-scale multilingual data. Chinese-first (ZhipuAI is Chinese). European languages supported but not the primary focus.

## API Surface

### 1. Python SDK (primary)

```python
from glmocr import GlmOcr, parse

# One-liner (MaaS mode, needs ZHIPU_API_KEY)
result = parse("document.pdf")

# Class-based
with GlmOcr(api_key="sk-xxx") as parser:
    result = parser.parse("image.png")
    print(result.markdown_result)
    print(result.json_result)  # structured with bbox
    result.save(output_dir="./output")
```

### 2. CLI

```bash
glmocr parse image.png
glmocr parse directory/
glmocr parse image.png --output ./results/ --stdout --api-key sk-xxx
```

### 3. Flask REST Server

```bash
python -m glmocr.server  # POST /glmocr/parse
```

### 4. FastAPI Backend (apps/backend/)

Full web app with task queue, SQLite DB, file upload. React frontend included.

### 5. Claude Code Skills

Ships 4 skill definitions (SKILL.md) for agent integration: general OCR, formula, table, handwriting specializations.

## Apple Silicon / macOS Support

**Documented and supported via MLX deployment**:
- Uses `mlx-vlm` for Metal GPU inference
- Requires macOS 14.0+ (Sonoma), Apple Silicon (M-series)
- Model: `mlx-community/GLM-OCR-bf16` from Hugging Face
- 0.9B params fits in 8GB unified memory
- Requires two Python environments (mlx-vlm and SDK have conflicting transformers versions)
- First request slow (Metal shader compilation), subsequent requests fast

**Ollama also works on macOS** with no GPU requirement (CPU inference).

## Comparison: GLM-OCR vs Apple Vision Framework

| Dimension | GLM-OCR | Apple Vision (VNRecognizeTextRequest) |
|-----------|---------|--------------------------------------|
| **Architecture** | VLM (vision-language model) | Traditional OCR + ML |
| **Model size** | 0.9B params | System framework, no download |
| **Formula recognition** | Yes (LaTeX output) | No |
| **Table recognition** | Yes (Markdown tables) | No (text only) |
| **Layout analysis** | Yes (25 region types) | Basic (text blocks) |
| **Handwriting** | Yes | Yes (good for English) |
| **Languages** | CJK + multilingual | 18+ languages |
| **Offline** | Yes (self-hosted/MLX/Ollama) | Yes (on-device) |
| **Speed** | Slower (VLM inference) | Fast (optimized on-device) |
| **Privacy** | Local possible, MaaS optional | Always local |
| **Platform** | Cross-platform (Python) | Apple only (Swift/ObjC) |
| **Integration** | Python/CLI/REST | Native Swift API |
| **Accuracy on complex docs** | SOTA (94.62 OmniDocBench) | Good for simple text |

**Verdict**: GLM-OCR is superior for complex documents (tables, formulas, mixed layouts). Apple Vision is better for fast, simple text extraction with zero setup. They are complementary, not competing.

## Shikki Integration Assessment

### For /ingest pipeline (OCR PDFs, images)

**HIGH VALUE**. The `/ingest` pipeline currently has no OCR capability. GLM-OCR would enable:
- Ingesting scanned PDFs, screenshots, whiteboard photos
- Extracting structured content (tables, formulas) from research papers
- Processing manga/comic page text extraction

**Integration path**: MaaS mode is simplest (just API key). For privacy/offline: Ollama or MLX deployment.

### For media reader (scan documents, manga, screenshots)

**MEDIUM-HIGH VALUE**. The structured output (JSON with bbox + labels) is excellent for:
- Document understanding (know where text, tables, images are on page)
- Screenshot-to-text for research
- Manga text extraction (vertical text supported)

### Can it run locally on macOS Apple Silicon?

**YES**, two paths:
1. **MLX** (recommended): Native Metal GPU, fast inference, 8GB RAM sufficient
2. **Ollama**: CPU or Metal, simpler setup, slightly slower

### Cloud dependency?

**NONE required**. MaaS mode is optional convenience. Full self-hosted operation is first-class.

### Blocking concerns

1. **Python-only SDK** -- Shikki is Swift. Integration requires either:
   - Shell-out to `glmocr` CLI (simplest, external tools philosophy)
   - Run Flask/FastAPI server + call via NetKit HTTP
   - Rewrite as MCP tool server (most aligned with ShikkiMCP)
2. **Two-env issue on MLX** -- transformers version conflict is annoying but manageable
3. **Alpha software (v0.1.4)** -- API may change
4. **Chinese-first project** -- some code comments in Chinese, docs bilingual

## Verdict

**ADOPT for /ingest, TRIAL for media reader**. GLM-OCR is the best open-source document OCR available today. The 0.9B model size makes it practical for local deployment on Apple Silicon. The Python SDK is well-designed with clean agent integration patterns. The two-stage pipeline (layout + VLM OCR) produces structured output that is far more useful than raw text. Main friction is Python-only SDK in a Swift codebase, but shell-out or HTTP server patterns solve this cleanly.
